1 SR&ED in 2 Minutes — Business Context
What is SR&ED?

Scientific Research and Experimental Development — Canada's largest government R&D support program. Companies that do qualifying technical work can claim a portion of their labour costs back as a tax credit from the CRA (Canada Revenue Agency).

Think of it as: "The government pays you back a percentage of what you paid your engineers to solve hard technical problems."

Who qualifies?
Canadian corporations doing R&D
Must be "systematic investigation" advancing technology or science — not just using existing tech
How big is the program?
$4.5B in credits per year
Over 21,000 companies claim annually. Canada's #1 R&D incentive.
Federal credit rate
35% refundable (CCPCs)
On first $6M of eligible expenditures. 15% above that. Refundable = cash back even if no profit.
Provincial top-up (ON)
~8–11.5% additional
ON: 8% OITC + 3.5%. BC: 10%. AB: ~10–20%. QC: up to 30%. Our app uses a combined rate.
How the money flows — the formula this app implements

1. Track hours per employee per project
2. Multiply by hourly rate (annual salary ÷ 2,000 hrs) → SR&ED expenditure
3. Multiply expenditure by SR&ED credit rate (e.g. 45% for Northwind Labs) → SR&ED credit
4. Unclaimed hours = hours on non-eligible projects (admin, ops, etc.) — not in the claim
5. Projection: extrapolate YTD rate over the full year → estimated annual credit

💡 Talking point: "The SR&ED credit rate in our mock data (0.45 for Northwind Labs) represents the combined federal + provincial effective rate for an Ontario CCPC."
What qualifies as SR&ED eligible work?
  • Basic/Applied Research — advancing scientific knowledge
  • Experimental Development — creating/improving products or processes using tech advancement
  • Support work — engineering, computer programming, testing, data collection directly supporting eligible work
  • NOT eligible: market research, quality control, routine testing, admin, styling/UX without tech uncertainty
💡 In the app: projects flagged isSredEligible: true (e.g. "Rendering System", "ML Inference Pipeline") count toward the claim. "Unclaimed Work" is false.

5 Technical Q&A Prep
Why Angular 21 with signals instead of RxJS?

Signals are Angular's new reactive primitive — synchronous, fine-grained, and predictable. All derived state in this app uses computed(): if you change the mode signal, every computed that depends on it (project bars, KPI, staff section, quarterly tabs) automatically re-evaluates. RxJS is reserved for async streams (HTTP calls) and bridged into the signal world with toSignal(). Zero BehaviorSubject, zero subscribe() for data flow in components.

Walk me through the RxJS in this project — what operators are you actually using?

Even though this is a signal-first app, RxJS lives at the HTTP boundary and the route layer. Concrete uses:

  • switchMap in the dashboard's tenantId → service.getAll(id) pipeline. If the user switches tenant mid-fetch, the in-flight request is cancelled and a fresh one starts. I chose switchMap over mergeMap because I don't care about the abandoned tenant's result — keeping it would risk a stale render.
  • toObservable / toSignal — the bridge between the two reactive worlds. toObservable(this.tenantId) turns my route-derived signal into a stream so I can pipe it through switchMap; toSignal(...) brings the result back into the signal world for the template.
  • startWith([]) — seeds the stream with an empty array so the signal has a defined value before the first emission. No undefined checks in the template.
  • map on the route's paramMap — pure projection from the ParamMap object to the tenant ID string.
  • of(...).pipe(delay(...)) in the mock services — shapes the mock response identically to a real http.get() Observable, so swapping mocks for HTTP is a one-line change per service.

All these services emit cold observables — each subscription triggers a fresh fetch. That's why switchMap can cancel cleanly without leaking subscriptions: the source isn't shared.

Decode this snippet — what's each operator doing, and where did ngOnInit / ngOnDestroy go?
readonly employees = toSignal(
  toObservable(this.tenantId).pipe(
    switchMap(id => this.employeesSvc.getAll(id)
      .pipe(startWith([] as readonly Employee[]))),
  ),
  { initialValue: [] as readonly Employee[] },
);

Shape: signal → toObservable → switchMap → startWith → toSignal. The two bridges sandwich an RxJS operator chain.

  • toObservable(this.tenantId) — signal-to-stream bridge. tenantId changes when the route changes; toObservable re-emits each new value so RxJS operators can react.
  • switchMap(id => ...) — on every new tenantId, cancel the in-flight request and start fresh. Fast tenant clicks A→B→C keep only C's response.
  • startWith([]) — emit [] immediately when the inner observable subscribes. On tenant switch, the signal drops to [] right away, which is what makes employeesReady flip back to false and the skeleton re-appear during the fetch.
  • toSignal(obs, { initialValue: [] }) — stream-to-signal bridge. initialValue is what the signal returns synchronously before any emission — without it, the type would be readonly Employee[] | undefined and the template would need null checks.

Where the lifecycle hooks went:

  • No constructor — inject() works in field initializers because Angular runs them during construction, when the injector is available.
  • No ngOnInit — the reactive graph is wired up at field-initialization time. By the time the constructor finishes, every signal is already listening to its source.
  • No ngOnDestroy — toSignal and toObservable auto-register cleanup with the component's DestroyRef. Subscriptions tear down when the component is destroyed.
  • For "code that must run after the view is in the DOM" (rare), use afterNextRender() — that's the modern ngAfterViewInit.

Old way vs new way — same semantics, ~half the code:

// OLD (RxJS + lifecycle hooks)
export class DashboardComponent implements OnInit, OnDestroy {
  employees: Employee[] = [];
  private destroy$ = new Subject<void>();
  private tenantId$ = new BehaviorSubject<string>('');

  constructor(private route: ActivatedRoute,
              private employeesSvc: EmployeesService) {}

  ngOnInit() {
    this.route.parent!.paramMap.pipe(
      map(p => p.get('tenantId') ?? ''),
      takeUntil(this.destroy$),
    ).subscribe(id => this.tenantId$.next(id));

    this.tenantId$.pipe(
      switchMap(id => this.employeesSvc.getAll(id).pipe(startWith([]))),
      takeUntil(this.destroy$),
    ).subscribe(emps => this.employees = emps);
  }

  ngOnDestroy() { this.destroy$.next(); this.destroy$.complete(); }
}

// NEW (signals + DestroyRef)
export class DashboardComponent {
  private readonly route = inject(ActivatedRoute);
  private readonly employeesSvc = inject(EmployeesService);

  private readonly tenantId = toSignal(
    this.route.parent!.paramMap.pipe(map(p => p.get('tenantId') ?? '')),
    { initialValue: '' },
  );

  readonly employees = toSignal(
    toObservable(this.tenantId).pipe(
      switchMap(id => this.employeesSvc.getAll(id).pipe(startWith([]))),
    ),
    { initialValue: [] },
  );
}
💡 One deliberate inconsistency to defend: client omits startWith(null), so the OLD client briefly persists during tenant switch — avoids flashing an empty header for ~200ms. The data signals use startWith([]) because the empty state is what we want (it triggers the skeletons). Different trade-offs per signal: stale-vs-flicker for the header, empty-vs-stale for the chart areas.

Verbal version for Xavier (~30 sec):

"It's a route-derived signal bridged into an RxJS pipeline. switchMap cancels in-flight on tenant change, startWith resets the data immediately so the skeleton shows, then toSignal brings the result back. The whole reactive graph is set up at field-initialization time — no ngOnInit. Subscriptions clean up via DestroyRef — no ngOnDestroy. Same logic I used to write with BehaviorSubject + takeUntil, declarative and half the code."

You use signals heavily, but SREDio is still on RxJS. Can you work with that?

Yes — and they're not mutually exclusive. This project actually uses both: RxJS at the HTTP and route layers, signals at the component layer, with toSignal / toObservable as the bridge. That's the modern Angular pattern, not a rejection of RxJS.

For a legacy RxJS codebase I wouldn't rip out BehaviorSubject and combineLatest on day one — they work fine and the team knows them. I'd introduce signals gradually as I write new components, using toSignal() to consume the existing observables. No big-bang rewrite.

There are places where RxJS still genuinely shines even in a signal-first app: anything async with operators that signals can't replace — debounced search input, retry-with-backoff, WebSocket streams, polling, animations sequenced with concatMap. Signals are synchronous; they don't replace those use cases.

One concrete example from this app: the login form's submit handler is the only subscribe() in the codebase. That's deliberate — it's an imperative side-effect (call login, then navigate), not a data subscription. RxJS is the right tool there.

Where exactly does RxJS still earn its keep, even in a signal-first app?

The clean rule: signals own state, RxJS owns async + events + time. Signals replaced state and derived state; they did NOT replace HTTP, events, or time-based operations.

RxJS is still the right tool for:

  • The HTTP layer. HttpClient returns Observable<T> — that's the producer. Every service in this project returns Observable. Signals only enter at the consumer side via toSignal.
  • Imperative side-effects with success/error — login, save, delete, submit. The subscribe({ next, error }) callback shape models a one-shot outcome with two branches. Signals model continuous state — wrong shape for "do this once, then react to the result."
  • Time-based operators that signals don't have: debounceTime (debounced search), retry({ count, delay }) (retry-with-backoff), throttleTime, interval + switchMap (polling), concatMap / mergeMap (sequential vs concurrent chains), animation sequencing.
  • Streams from external sources: WebSocket via webSocket(), Server-Sent Events, Router.events, route.paramMap, formControl.valueChanges, form.statusChanges. Consume them via toSignal if you want the latest value as a signal — but the source is RxJS.

The decision matrix:

DomainTool
State the template rendersSignal
Derived state (calculated from other state)computed()
Component inputs/outputsinput(), output()
HTTP responsesObservable → toSignal() at the consumer
One-shot side-effects (login, save, delete)subscribe({ next, error })
Time-based async (debounce, retry, poll)RxJS operators → toSignal() if you need the latest
External streams (WebSocket, SSE)RxJS, bridged at the boundary
💡 If pushed: "But signals have effect() — can't you do side-effects with that?" Answer: effect() runs when a signal changes — for reactive side-effects (e.g., persist a setting to localStorage when it changes). subscribe() runs when a stream emits — for one-shot or stream-driven side-effects. For a login POST: the click is the trigger, the HTTP response is the stream. subscribe() is right. effect() would require materializing the response into a signal first, then watching it — pointless ceremony.
If we were still on RxJS, what would the dashboard's reactive code look like? Show me the equivalent.

Best concrete example: projectBars on the dashboard. It depends on 5 sources (filtered entries, employees, projects, mode toggle, client's credit rate) and re-derives whenever any of them changes. Side-by-side:

Signal version (current code):

readonly mode = signal<SredMode>('hours');
onModeChange(m: SredMode): void { this.mode.set(m); }

readonly projectBars = computed(() =>
  projectBarData(
    this.periodEntries(),
    this.employees(),
    this.projects(),
    this.mode(),
    this.client()?.sredCreditRate ?? 0.45,
  ),
);
<!-- template -->
<app-sred-projects-bar [bars]="projectBars()" />

Pure-RxJS equivalent (no signals):

private modeSubject = new BehaviorSubject<SredMode>('hours');
mode$ = this.modeSubject.asObservable();
onModeChange(m: SredMode): void { this.modeSubject.next(m); }

// assumes periodEntries$, employees$, projects$, client$ are all
// BehaviorSubjects or replayed observables already
projectBars$ = combineLatest([
  this.periodEntries$,
  this.employees$,
  this.projects$,
  this.mode$,
  this.client$,
]).pipe(
  map(([entries, employees, projects, mode, client]) =>
    projectBarData(entries, employees, projects, mode, client?.sredCreditRate ?? 0.45)
  ),
  shareReplay({ bufferSize: 1, refCount: true }),  // multicast for multiple template subscribers
);
<!-- template -->
<app-sred-projects-bar [bars]="projectBars$ | async" />

What's different and why signals are nicer:

  • No combineLatest boilerplate. Signals track their dependencies automatically — you just read this.mode() inside computed() and it gets wired up. With RxJS you have to list every source explicitly and remember to add new ones when the formula changes.
  • No shareReplay required. Signals are inherently multicast — every consumer reads from the same memoized value. With combineLatest, every | async in the template subscribes independently and re-runs the whole pipeline unless you shareReplay.
  • Synchronous read. this.projectBars() returns the value right now — no .subscribe, no race against initial emission. RxJS users handle "no value yet" with startWith + BehaviorSubject + initial defaults at every consumer.
  • No .next() for state changes. this.mode.set('credits') vs this.modeSubject.next('credits'). Cosmetic, but signals don't expose the Subject — there's no way for external code to push values, only the component-defined setters/methods.
💡 What stays the same: the underlying logic — projectBarData(entries, employees, projects, mode, creditRate) is a pure function called identically in both versions. That's the part you want to lift into features/<X>/calculations/ regardless of reactive style. Pure functions outlive any framework decision.

Verbal version:

"In a pure-RxJS world this projectBars would be a combineLatest of five source observables, mapped through the same projectBarData() pure function, with shareReplay so the template's | async consumers don't each trigger a fresh pipeline. The signal version is a computed() that reads the same five sources synchronously — automatic dependency tracking, automatic multicasting, no shareReplay, no startWith, no BehaviorSubject ceremony. The underlying calculation function is identical in both. That's the whole point of keeping the math as pure functions in features/<X>/calculations/ — the reactive layer is swappable, the math isn't."

How would you migrate an RxJS-heavy codebase to signals incrementally?

Same Strangler Fig discipline as the Material → Flowbite migration: incremental, leaf-first, never big-bang.

  1. Audit first. Inventory every BehaviorSubject, combineLatest, withLatestFrom, bare subscribe(), and | async in the codebase. You can't plan what you haven't measured.
  2. Leaf components first. Pick a component that consumes an observable but doesn't produce one. Wrap with toSignal(), drop the | async in the template, ship. Lowest risk, highest pattern-establishing value.
  3. Replace BehaviorSubject + .getValue() with signal(). Same shape — a holder of a value you can read synchronously. If anything upstream still needs an observable, toObservable(mySignal) covers the gap.
  4. Replace combineLatest(...).pipe(map(...)) with computed(). Cleaner, synchronous, no subscription management, no race conditions on initial value.
  5. Keep RxJS at the boundaries. HTTP layer, WebSocket layer, anywhere with retry/debounce/polling. Bridge to signals at the component edge with toSignal().
  6. Treat subscribe() as a code smell. Rare exceptions: imperative side-effects (login → navigate), fire-and-forget analytics. Always pair with takeUntilDestroyed() — never bare.
💡 Common pitfalls to flag:
  • toSignal() without initialValue produces T | undefined — always provide one for component-bound signals so the template doesn't need null checks.
  • Hot subjects (Subject, not BehaviorSubject) can drop emissions before toSignal subscribes — late subscribers get nothing. Use shareReplay(1) or convert to BehaviorSubject first.
  • effect() and subscribe() aren't interchangeable — effect runs synchronously on signal changes during the reactive phase; subscribe runs async on stream emissions. Pick deliberately.
What is zoneless change detection and why use it?

Angular traditionally uses Zone.js to monkey-patch browser APIs and trigger change detection on every async event. With provideZonelessChangeDetection(), Zone.js is removed entirely. Change detection only runs when a signal changes — much more efficient. Combined with OnPush on every component, Angular only re-renders the components whose signal inputs actually changed.

Why pure functions for SR&ED calculations instead of a service?

Services imply DI, lifecycle, and often hidden state. These calculations have none of that — they're just math: inputs in, output out. Pure functions in features/dashboard/calculations/ are trivially unit-tested (no mocking, no TestBed), easily composed, and impossible to misuse. staffBarData(entries, employees, projects, mode, creditRate) is self-documenting.

How would you swap the mock data for a real API?

Every service has two code paths behind an environment.useMocks flag: the mock branch returns of(mockData).pipe(delay(300)), the real branch does http.get<T>(url). The Observable shape is identical — components never know the difference. The swap is one-line per service. Guards and interceptors are already real implementations.

Explain the asOf signal and why it's a computed, not a constant.

asOf represents "today relative to the selected claim period." For the live 2026 period, it's today (2026-05-19) — so YTD stops at today. But for a past period like 2025, using today as the cutoff would pull in 2026 entries. The fix: asOf = computed(() => min(CURRENT_DATE, activeClaimPeriod.endDate)). Past periods use their own end date (Dec 31); the current period uses today. CURRENT_DATE is still a constant — it's the raw "today" the computed reads from.

Where do you draw the line between frontend and backend for data calculations?

I think about it as source-of-truth + auditability. Backend owns anything that gets filed, persisted, or audited; frontend owns view-level derivations and interactive pivots.

Backend owns:

  • Any number that ends up in a tax filing or ledger (the actual SR&ED credit submitted to CRA)
  • Expensive aggregations across large datasets
  • Anything multiple clients need to agree on (web, mobile, scheduled email, PDF export) — three clients reimplementing the math means three chances to drift
  • Anything dependent on data the FE shouldn't see (access control, cross-tenant rollups)

Frontend owns:

  • View pivots — raw entries → bar chart bins
  • Anything driven by instant user interaction (the Hours/Expenditures/Credits toggle, drill-downs, filters)
  • Formatting, sorting, currency display

Litmus test: "If this number went on a tax form, would I trust the frontend to compute it?" If no → backend.

💡 Defending calculations/project-bar-data.ts: it's a view pivot, not the canonical SR&ED total. With a mode toggle that re-pivots on every click, round-tripping each toggle would feel terrible. Backend ships raw entries + the canonical claim total; frontend derives the chart bars locally.
If both backend and frontend compute the same number and they disagree, whose fault is it?

The discrepancy itself is the bug. The right answer isn't to assign blame — it's to never have two implementations of the same calculation.

  • Single source of truth: backend is canonical for any number that ends up in a filing. Frontend re-derives only the presentation of that data, never the official total.
  • If both must compute the same number: share the code — a TypeScript package both Node BE and Angular FE consume — or write contract tests against golden inputs/outputs so drift surfaces immediately.
  • Triage rule in production: backend wins. If the dashboard shows $52k but the CRA filing says $48k, the $48k is what was actually filed — the bug is on the frontend (stale cache, wrong date filter, missing employee). You debug the divergence, you don't argue about which is "right."

Defense in depth: even with backend as canonical, I'd write a small set of golden tests — fixed input → expected output — that run on both sides. Catches the drift before users do.

How would you structure API calls for a dashboard like this — one big call or one per card?

Three real patterns, each with tradeoffs:

  • A. One coarse "page payload" endpoint (GET /tenant/:id/dashboard?period=2026) — internally consistent snapshot, simple FE, one auth check. But slow if any one section is slow, and you can't cache per-card.
  • B. Per-section endpoints (/summary, /projects, /staff…) — parallel load, granular caching, lazy-loading. But cards may show inconsistent snapshots mid-load, and you maintain more endpoints.
  • C. GraphQL / BFF — frontend declares exactly what it needs, backend assembles. Best of both, but operational complexity. Worth mentioning only if Xavier brings it up.

What I'd actually do for a dashboard: a hybrid — coarse-grained, not one-per-card. Typically:

  • One cheap "summary" call — claim period, status, headline KPIs. Loads in <200ms, page is interactive immediately.
  • One heavier "details" call in parallel — raw entries, employees, projects — fills in the charts as it arrives.
  • Drill-down calls on demand — clicking an employee fetches their per-day timesheet. Not loaded upfront.

For this project: the mock services use pattern A — one tenant payload per dashboard load — which is fine at this scale. In production I'd split timeEntries[] into a lazy call once it grows past a few thousand rows.

Caching as a senior nuance: backend returns Cache-Control + ETag; frontend caches by (tenantId, periodId) — this app does it implicitly via signals. "Recalculate" issues a conditional request — most cards re-render from cache if nothing changed.

How does multi-tenant routing work?

Every authenticated route is under /tenant/:tenantId/. The tenant ID flows from the route into every service call via a toSignal(route.parent.paramMap) in the dashboard component — reactive to navigation. A tenant guard validates the ID exists and redirects to the last-used tenant or first tenant on login. Switching tenants is just a router.navigate(['/tenant', id, 'dashboard']) — no page reload, just reactive data refresh.

What would you build next if given more time?
  • Export to PDF — CRA submissions need paper trails; ApexCharts SVG makes this tractable
  • Per-employee eligibility % — currently a project is 100% or 0% eligible; real CRA claims have fractional allocations
  • Backend integration — replace the one-line mock swap with real HTTP + auth token refresh
  • Vendor invoices / Other Expenses section — the reference UI showed this; it's the next claim section
  • Weighted projection model — linear extrapolation is naive; weight recent quarters, account for seasonality
What is the SR&ED credit rate in the app and where does it come from?

Each Client object has a sredCreditRate property (e.g. 0.45 for Northwind Labs / ON). It represents the combined effective federal + provincial rate for that client's province. The formula: SR&ED credit = SR&ED cost × creditRate. In the Credits mode, this is what all the numbers show. If the rate is missing, the app defaults to 0.45.

The Key Insight: Two Very Different Types of Migration

Angular version upgrade (what went wrong before) → forced big bang. You can't run Angular 12 and Angular 16 in the same app simultaneously. The whole codebase has to move together. The lesson there wasn't "big bang was wrong" — it was "we didn't audit deeply enough first."

Angular Material → Flowbite/Tailwind (this job) → Strangler Fig is possible. Both UI libraries can coexist in the same Angular app at the same time. You can migrate one page, one component at a time. This is the right approach here and you can explain it confidently.

The Strangler Fig Pattern — How to Explain It
1
Audit first (1–2 weeks, no code written)
"Before touching anything, I'd do a full audit: which Angular Material components are used, how many instances of each, which pages are highest-risk. This is the step I skipped in my previous migration and it hurt us. You can't plan what you don't understand."
2
Set up Tailwind + Flowbite alongside Angular Material
"Tailwind and Angular Material are not mutually exclusive — they can both be active in the same project. I'd add Tailwind to angular.json styles, configure PostCSS, and install Flowbite. Nothing breaks. We now have both available."
3
Build Flowbite replacement components in parallel
"For each Material component we use, I'd build a standalone Flowbite equivalent — same @Input()/@Output() API where possible so swapping it in a template is a one-line change. These live alongside the Material components until we're ready to swap."
4
Migrate page by page — lowest risk first
"We start with the simplest, least-used pages. Ship, test, move on. We never touch a form-heavy page or mission-critical flow until we have a clear pattern established on simpler pages."
5
Complex components last: forms, dialogs, date pickers
"MatFormField and MatInput are the trickiest — forms touch almost every page. I'd handle these last, once we have momentum and confidence. MatDatepicker needs a third-party replacement since Flowbite doesn't ship one."
6
Remove Angular Material only at the very end
"Angular Material is only removed from package.json once every component has been replaced. By that point the risk is zero — we've already tested everything. The old library just stops being loaded."

How do you frame your previous migration experience?

Don't hide it — Xavier will respect honesty far more than a perfect story. Say this:

"I did an Angular major version upgrade at a previous company — that's a forced big bang, you can't do it incrementally. It was painful because we had a large, spaghetti codebase, no test suite, and custom CSS classes that weren't caught until they broke in production. I'd never start a migration without first mapping every component, every custom class, every page that might be affected. For Angular Material → Flowbite specifically, the Strangler Fig pattern also means we don't have to big bang at all — which eliminates most of the risk we had before."

What if SREDio is on Angular 16 or lower? Would you upgrade Angular while migrating to Flowbite?

Key fact: the official Flowbite install is npm install flowbite — the vanilla JS package. It has no Angular peer dependency on any version. Verified on npm: flowbite@4.0.2 peerDeps are empty, only Tailwind 4 is a hard dep. So the Flowbite migration is decoupled from the Angular version — it works on Angular 12, 16, 19, 21, anything.

That means the two are independent workstreams, not one bundle:

  • Flowbite migration: can start on whatever Angular version is current. The Angular component wrapper uses whatever API the codebase already uses — @Input/*ngIf on v16, input()/@if on v18+. The Tailwind HTML and Flowbite JS init are identical either way.
  • Angular upgrade: separate workstream, motivated by modern idioms (signals, control flow, standalone), not by Flowbite.

Order (if no business pressure): Angular upgrade first, then Flowbite. Writing new components with signal(), input(), and @if is cleaner than wrapping Flowbite in old-style decorators and refactoring later. The "refactor later" is real but small — only the Angular wrapper API changes; Tailwind classes and Flowbite JS calls stay identical.

Sweet spot for the Angular landing version: v18 or v19, not v21.

  • v17: control flow and signal inputs still in developer preview — would rather skip as a landing target
  • v18: control flow stable, input()/output()/model() stable, zoneless experimental — minimum I'd want
  • v19: standalone is the default, less boilerplate — even better
  • v20 → v21: small later hops once Flowbite work is done. No need to chase v21 just to migrate UI.

Fallback framing if business says "we need Flowbite this quarter": "Then we start Flowbite on the current Angular version. Components written today will need a light wrapper-API refactor after the Angular upgrade — Tailwind and Flowbite parts don't change. I'd write them forward-compatibly: standalone is available back to v15, toSignal back to v16, so we can lean into modern patterns even on an older version."

💡 Don't confuse flowbite with flowbite-angular. The latter is a community Angular-wrapper package that does pin to Angular major versions (flowbite-angular@1.3.0 requires Angular 18–19; v20/v21 require matching majors). It's not what the official docs recommend and not what this project uses. If Xavier asks specifically about flowbite-angular, the version-lock is real and worth flagging.
What if they're already on Tailwind v3? Upgrade to v4 first, or stay on v3 during the Flowbite migration?

The hard constraint: flowbite@4.x requires tailwindcss@^4.1.12. flowbite@3.x works fine on Tailwind 3. So the real choice is: Flowbite 3 on Tailwind 3 (no Tailwind upgrade) vs Flowbite 4 on Tailwind 4 (upgrade first).

Recommendation: upgrade Tailwind 3 → 4 first, then bring in Flowbite 4. Reasoning:

  • Tailwind 4 has an official codemod — npx @tailwindcss/upgrade — that handles most of the mechanical changes (CSS-first config, renamed utilities like shadow-sm semantics, flex-shrink-0 → shrink-0, opacity modifiers).
  • If we migrate to Flowbite 3 on Tailwind 3 today, every component we ship needs another touch when Tailwind 4 arrives — that's writing the same Tailwind classes twice in different syntaxes.
  • Flowbite 4 has a year of fixes and component additions over 3.x. Better foundation for new work.
  • Tailwind 4 uses Lightning CSS — much faster dev builds, smaller output.

When I'd flip the recommendation (stay on Tailwind 3 + Flowbite 3):

  • Heavy tailwind.config.js customization (custom plugins, deep theme extensions) — the v4 CSS-first config migration becomes its own project
  • Browser support requirement below Safari 16.4 / Chrome 111 / Firefox 128 — Tailwind 4 drops older browsers
  • Tight timeline where Tailwind 4 audit work would delay the actual Flowbite migration

Always audit first. Same lesson as the Angular upgrade — run the codemod on a branch, see what breaks, scope the work. A clean codebase is a one-day Tailwind upgrade. A heavily-customized one might be two weeks.
